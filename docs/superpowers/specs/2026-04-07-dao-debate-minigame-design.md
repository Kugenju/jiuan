# 道法论辩小游戏设计

## 目标

在当前多周校园修仙 demo 中，为道法方向新增一套“论辩型限时任务”玩法。该玩法需要：

- 复用现有通用限时任务框架，而不是另起一套独立系统
- 让玩家感受到“这一周的课程与见闻，是否真的转化成了可被追问、可被拆招的论证能力”
- 与炼器任务形成明显差异：炼器偏实践解题，道法偏短回合攻防
- 作为后续更多道法论题与其他思辨类小游戏的首个模板

## 范围

本次设计包含以下内容：

- 道法任务的触发、排程、重试、过期与奖励规则
- 三轮论辩攻防的局内闭环
- 论点牌、追问类型、`conviction` / `exposure` 判定
- 隐藏论点牌与外层课程/事件的接法
- 首版题目、首版牌池与 UI 信息结构

本次设计不包含以下内容：

- 十道完整论题的全部内容扩展
- 自然语言输入或自由文本作答
- 4 轮以上长局
- 动态生成问题或 AI 即时追问
- 多标签叠乘、复杂连锁状态与高复杂度平衡

## 背景设定

太学院道法阁中有一尊喜与学员论道的灵偶，为前任太学祭酒少年时所作。这尊灵偶陪伴前祭酒游历四方，见证过诸多传奇旧事。待其成长渐渐跟不上主人脚步后，便被留在道法阁中，替书院看守玉筒灵牍。

然而这尊灵偶早年见闻太广，不甘终日与死物为伴，常拉着来此查阅典籍的师生论道。师生们初时不堪其扰，久而久之却也承认他见识广、反应快、善拆人理路。于是道法院长老顺势让其负责考核学员的“论道素质”。因它每每听到妙处总爱连声赞叹“妙哉、妙哉”，学员便私下称其为“妙哉偶”。

设定层保留“妙哉偶有十道论题”的说法，但 demo 首版仅落地第一道。

## 设计意图

道法小游戏不应退化为：

- 普通答题找标准答案
- 单纯剧情分支选对选项
- 纯文本阅读后拿固定奖励

它要测试的是：玩家面对价值冲突时，能否提出一套经得起三轮追问的论证。

因此首版的核心体验定为：

- 玩家先表明立场
- 妙哉偶根据玩家上一轮采用的论法发起针对性追问
- 玩家继续补强、转圜或拆解
- 三轮结束后判断这套说法是否站得住、是否暴露出明显破绽

## 方案选项

### 方案 A：标准答案型问答

每题预设若干答案，累计隐含分过线即通过。

优点：

- 最容易实现
- 文本量可控

缺点：

- 更像普通分支对话
- 缺乏“论辩攻防”感
- 难以与隐藏论点、课程积累形成结构性联动

### 方案 B：论点资源攻防（采用）

玩家操作对象不是整句台词，而是“论点牌”。妙哉偶的追问根据上一轮论点牌的标签变化，局内胜负同时看论证强度与破绽暴露。

优点：

- 明确体现“立论 -> 被拆 -> 再应对”的论辩体验
- 规则可数据化，不依赖复杂剧情树
- 易于与外层养成、隐藏牌、后续扩题兼容

缺点：

- 需要额外的数据结构与结算规则
- 首版需要克制牌型和状态数量

### 方案 C：开放输入或重剧情树

让玩家直接输入回答，或为每个回答编写完整追问分支树。

优点：

- 表现力上限高

缺点：

- 规则不可控
- 内容生产量过大
- 不适合当前 demo 阶段

## 已确认方向

- 采用限时任务外壳
- 胜负核心为“攻防博弈型”
- 玩家在局内操作“论点资源”而非纯选项文本
- 起手资源采用“小池半随机抽取”
- 追问变化采用“固定题干 + 按上一轮论点标签切换追问类型”
- 首版固定为三轮短局，不做长局、不做中途即时失败

## 通用任务接法

### 任务类型

- 任务类型：`dao_debate`
- 活动 ID：`dao_debate_task`
- 每周最多生成 1 次

### 触发规则

- 当周完成一定数量的道法课程后，触发妙哉偶注意到你的提示
- 首版建议解锁条件为：当周 `dao` 课程完成数 `>= 2`
- 解锁提示在课程结算当下出现，但任务不立即进入可排程列表
- 到次日开始时，`道法论辩` 才加入自由时段候选活动

采用“次日可排程”的原因：

- 与炼器任务的即时实践感形成差异
- 更符合“今日被注意，明日受邀论辩”的叙事节奏
- 给本周事件解锁隐藏牌留出缓冲空间

### 排程与时限

- 玩家需主动将 `道法论辩` 安排进自由时段
- 每次执行任务都会占用当前时段
- 任务时限固定为 7 天，本质上受当周周期约束
- 任务到期未完成则失败并移除

### 尝试与重试

- 每次进入任务都重开一局，不继承局内进度
- 单次失败只消耗当前时段，不直接终止整项任务
- 只要未过期，玩家就能再次安排并重试

## 局内玩法闭环

每局固定 3 轮，每轮流程如下：

1. 妙哉偶提出主问题或追问
2. 玩家从当前手牌中打出 1 张论点牌
3. 系统根据“当前追问类型 + 本轮论点标签”进行结算
4. 系统根据本轮牌的标签决定下一轮追问类型

第 1 轮为主问题，第 2、3 轮为针对上一轮论法的追问。

首版不在中途判定“立刻失败”，而是在 3 轮结束后统一结算。这样玩家能完整体验一场论辩，而不是第一轮答歪就被强行中断。

## 论点牌设计

### 论点牌定位

每张论点牌代表一种“回应策略单位”，而不是一整段完整台词。玩家打出的其实是论法，而非逐字逐句回答。

### 首版标签

首版每张牌只带 1 个主标签，避免多标签规则过重。建议使用以下 6 类：

- `principle`：守义、守本、立道
- `utility`：衡量结果、取利避害
- `authority`：引经典、引师承、引先例
- `experience`：举亲历、举现实个案
- `counterexample`：举反例、拆对方假设
- `evasion`：回避核心矛盾、含混不决

这些标签仅作为系统判定依据。对玩家展示时，应以更有文气的牌名与简短释义包装。

## 追问变化机制

### 设计原则

追问变化不依赖复杂 AI，也不按每张具体牌名写剧情树。首版采用“上一轮标签 -> 当前追问类型”的映射表驱动。

### 追问类型

首版建议收敛为 4 类追问：

- `press_principle`：逼问你讲的“义”到底站不站得住
- `press_utility`：逼问你讲的“利”是不是只算眼前
- `press_authority`：逼问你是不是只会搬用前人之言
- `press_evasion`：逼问你是不是在回避核心取舍

### 推荐映射

- `principle -> press_principle`
- `utility -> press_utility`
- `authority -> press_authority`
- `evasion -> press_evasion`
- `experience -> 由题目预设映射到 press_principle 或 press_utility`
- `counterexample -> 首版并入 press_principle`

这样可以用较少的追问模板，覆盖更多出牌情况，并保持系统感清晰。

## 胜负判定

### 双值模型

局内同时记录两个核心值：

- `conviction`：论证站得住的程度
- `exposure`：被妙哉偶抓住的破绽数

### 每轮结算

- 强应对：`conviction +2`
- 可接受应对：`conviction +1`
- 弱应对：`exposure +1`
- 若打出 `evasion`：默认 `conviction +0`，且 `exposure +1`
- 若连续两轮使用弱势逻辑，例如连续 `authority` 或连续 `evasion`，追加 `exposure +1`

### 最终判定

- 通过：`conviction >= 5 && exposure <= 1`
- 勉强通过：`conviction >= 4 && exposure == 0`
- 失败：其余情况

这样玩家不能只追高分，还必须控制破绽，才能体现论辩攻防而不是普通累计答题。

## 起手与牌池

### 起手规则

- 每局从本局牌池中半随机抽取 5 张作为起手
- 3 轮内各打 1 张
- 首版不补牌、不换牌

这种 `5 选 3` 结构能让玩家在“当前最优回应”和“保留后手”之间做取舍，同时维持可控复杂度。

### 首版牌池结构

首版建议总池 8 至 10 张：

- 基础常驻牌 4 张：保证任何一局都能跑通
- 方向牌 3 至 4 张：覆盖原则、结果、权威、经验、反例等方向
- 隐藏牌 1 至 2 张：平时不入池，满足条件后才加入

基础常驻牌中应尽量覆盖：

- 1 张稳健牌
- 1 张高风险高收益牌
- 1 张补强牌
- 1 张保底牌

## 隐藏论点牌

### 接法

原始设想中的“隐藏选项”，在实现上正式收敛为“隐藏论点牌”。

其流程为：

- 某些课程、事件或人物互动给玩家提供 `unlockFlag`
- 本周生成 `dao_debate_task` 时，根据已有 `unlockFlag` 将对应隐藏牌加入本周任务池
- 玩家进局后，有机会在起手 5 张里抽到这张牌

### 设计原则

- 隐藏牌的强度应体现在“能破解某类追问结构”，而不是简单更高分
- 每张隐藏牌只强在一个明确方向
- 首版每局最多允许 1 张隐藏牌入池
- 不做万能正确答案，否则玩法会退化为“拿钥匙开门”

## 首版题目

### 设定层

- 妙哉偶有十道论题
- demo 首版仅启用第一题

### 第一题建议

- 题名：`术可代德否`
- 主问题：`若一门术法可救万人，却需修者常年损德折寿，此术当兴还是当禁？`

选择这道题的原因：

- 冲突明确，易于理解
- 可同时容纳原则、结果、经典、经验、反例等论法
- 容易形成连续追问

## 奖励与失败反馈

### 成功奖励

首版奖励保持克制，避免任务变成高收益刷分点。建议包含：

- `dao +1`
- `insight +1` 或少量 `spirit`
- 周总结中的“论道过关”标记

若是优秀通过，首版优先体现在文本反馈质量上，不急于放大数值差。

### 失败反馈

- 单次失败：消耗当前时段，记录“论证未稳，被妙哉偶指出破绽”
- 到期失败：任务移除，并在周总结中留下一条明确负反馈

首版不建议直接扣属性。

## UI 信息结构

首版 UI 需要清楚呈现以下内容：

- 当前题干或追问文本
- 当前为第几轮
- 玩家当前手牌
- `conviction`
- `exposure`
- 任务剩余天数
- 妙哉偶对本轮应对的即时评价

建议主舞台上让主角与妙哉偶分列两侧，中央承载题干/追问，下方展示论点牌，以强化“论辩对峙”感。

## 状态与数据建议

### 任务实例

建议至少包含：

```js
{
  id: "week-2-dao-debate",
  type: "dao_debate",
  activityId: "dao_debate_task",
  status: "active",
  unlockDay: 3,
  availableFromDay: 4,
  expiresOnDay: 7,
  attemptCount: 0,
  topicId: "topic_1",
  rewardClaimed: false,
}
```

### 局内运行态

建议至少包含：

```js
{
  taskId: "week-2-dao-debate",
  topicId: "topic_1",
  roundIndex: 0,
  hand: [],
  playedCards: [],
  currentPrompt: null,
  currentFollowupType: null,
  conviction: 0,
  exposure: 0,
  result: null,
}
```

### 数据配置

建议为以下内容保留独立配置层：

- 论题定义
- 论点牌定义
- 追问映射规则
- 判定关系表
- 隐藏牌与 `unlockFlag` 对应关系

## 测试策略

至少覆盖以下三组测试：

### 任务流程测试

- 当周道法课程达到解锁条件后正确触发提示
- 任务在次日才进入可排程列表
- 单次失败后可再次排程
- 到期未完成会失败并移除
- 每周不会重复生成多次

### 局内规则测试

- 起手抽牌数量正确
- 三轮流程正确推进
- 追问按上一轮标签切换
- `conviction` / `exposure` 结算正确
- 连续弱势逻辑时会追加破绽

### 接线测试

- 从自由时段执行任务后能正确进入 `task` 模式
- 局内结算后能回到日程推进
- 周总结能读取成功或失败标记

## 风险与约束

- 若牌池和标签过多，首版会很快失去可读性
- 若隐藏牌做成万能解法，会削弱论辩本身的学习感
- 若追问类型继续细分过多，文本成本会快速膨胀
- 若 UI 同时暴露太多隐藏规则，玩家会把系统理解成算表，而非论辩

因此首版必须坚持：

- 三轮
- 单标签牌
- 少量追问类型
- 小牌池
- 克制奖励

## 实施建议

先完成通用任务接线与首版一题最小闭环，再补充更多题目、牌型和隐藏牌来源。首版优先验证的是：

`道法课程触发 -> 次日开放排程 -> 三轮论辩 -> 通过/失败 -> 周循环反馈`

只要这条链路稳定成立，道法小游戏就具备继续扩展的基础。
