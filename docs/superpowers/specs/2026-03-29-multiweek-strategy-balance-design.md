# 多周策略验证与长期代价设计

## 背景

当前 demo 的主循环已经支持：

- 一周固定课表选择
- 每天自由时段安排
- 白天活动结算
- 夜间灵块放置与建筑结算
- 周结算评级

但目前仍然是单周结构：

- `totalDays` 固定为 `7`
- 第 7 天夜晚后直接进入 `summary`
- 周结算文案与界面仍然写死为“第一周”
- 自动实验脚本也只跑到第一次 `summary`

这会导致数值验证只看得到“单周强不强”，看不到“同一种策略连续使用 2-4 周后是否开始失衡”。

## 目标

本次设计要支持“固定同一套课表，连续 3-4 周测试自由时段策略的长期代价”，并满足下列体验目标：

1. `全课业`、`全打工`、`全修炼` 这类极端策略在第 1 周要有亮眼表现。
2. 连续 2-3 周重复同一极端策略后，要出现明显但可理解的惩罚。
3. `均衡混排` 在长期测试中应具备追平或反超极端路线的可能。
4. 周结算必须能解释“为什么这一周强/为什么后续开始变钝”。
5. 完成数值调整后，需要把最终落地规则同步进 [策划案整合.md](/F:/personal/game_t/xianDemo/策划案整合.md)。

## 非目标

本轮不处理以下内容：

- 重做课程选择逻辑
- 改变固定课表的生成方式
- 深入重做夜间建筑系统的数值
- 引入新的大类活动路线
- 一次性扩展到学期级、学年级长期运营

## 当前周结算评价标准

当前周结算分两层：

### 1. 先计算隐藏分

在 [src/domain/summary.js](/F:/personal/game_t/xianDemo/src/domain/summary.js) 中，当前会取以下维度：

- `resources.coins`
- `resources.insight`
- `resources.spirit`
- `stats.aura`
- 全技能总和

处理方式：

1. 先按当前已知上限做归一化
2. 再取平方根，形成边际递减
3. 再按权重加总
4. 再加一个“第三强核心维度奖励”

当前权重大意如下：

- 灵石 `7`
- 灵感/悟道 `7`
- 灵力 `6`
- 灵压 `4`
- 技能总和 `5`
- 第三维奖励 `8`

其设计意图是：

- 避免单一指标无限统治评分
- 让至少三维不短板的路线更稳定

### 2. 再映射到品级

在 [data/story.js](/F:/personal/game_t/xianDemo/data/story.js) 中，当前阈值为：

- `28+` 上上品
- `24+` 上中品
- `20+` 上下品
- `17+` 中上品
- `14+` 中中品
- `11+` 中下品
- `8+` 下上品
- 其余 下中品

## 核心设计

### A. 单周终局改为多周阶段结算

保留“同一套周课表固定沿用”的前提，把当前一次性单周流程改为多周循环：

1. 第 1 周开局时选择原型与课程
2. 生成固定周课表
3. 第 1-4 周都沿用这份周课表
4. 每周第 7 天夜晚结束后进入“第 N 周结算”
5. 结算后更新长期代价状态
6. 未达到总周数时进入下一周第 1 天
7. 最后一周结束后再进入总评界面

这意味着：

- `summary` 不再只是整局终局，而是“周报”
- 需要额外新增一个“最终总评”阶段，或让最后一次 `summary` 同时展示周报与总评

### B. 新增多周状态

建议在根状态中新增以下字段：

- `week`
- `totalWeeks`
- `dayOfWeek`
- `weeklyReports`
- `routeStress`
- `strategyHistory`

建议语义：

- `week`：当前周序号，起始为 `1`
- `totalWeeks`：本轮测试总周数，默认 `4`
- `dayOfWeek`：当前周内第几天，范围 `1-7`
- `weeklyReports`：每周结算快照数组
- `routeStress`：长期代价状态
- `strategyHistory`：每周主路线与占比记录

其中 `routeStress` 结构建议为：

```js
{
  study: 0,
  work: 0,
  training: 0
}
```

范围限制：

- 每项 `0 - 6`

### C. 主路线识别规则

只统计“自由时段中由玩家手动填入的活动”，不统计固定课程。

路线映射建议如下：

- `study`：`homework`
- `work`：`part_time`
- `training`：`training`
- 其余活动视为缓冲类，不计入极端路线透支

每周结束后统计自由时段占比：

- 如果某一路线占比 `>= 50%`，且明显高于第二名，则判为本周主路线
- 否则判为 `balanced`

这条规则用于避免：

- 均衡路线被误判为某个极端流派
- 固定课程过多时干扰自由时段策略识别

### D. 长期代价：`routeStress`

`routeStress` 的设计目标不是惩罚某一天，而是惩罚“连续几周都用同一种自由时间解决问题”。

每周结算后更新，作用于下一周。

建议更新规则：

- 本周为 `balanced`
  - `study/work/training` 全部 `-1`
- 本周主路线为 `study`
  - `study +2`
  - `work -1`
  - `training -1`
- 本周主路线为 `work`
  - `work +2`
  - `study -1`
  - `training -1`
- 本周主路线为 `training`
  - `training +2`
  - `study -1`
  - `work -1`
- 若本周主路线与上周相同
  - 对应路线额外 `+1`

最终统一 clamp 到 `0 - 6`。

预期节奏：

- 第 1 周极端路线无历史包袱，表现亮眼
- 第 2 周重复同一路线，开始变钝
- 第 3 周再重复，惩罚显著
- 切换到均衡路线后，代价能逐周恢复

### E. 惩罚生效规则

长期代价只打击该路线最核心的正反馈，不做全局盲扣。

#### `study` 透支

影响对象：

- `homework`
- 必要时可轻微波及学习类自由活动的额外收益

建议档位：

- `0-1`：无惩罚
- `2-3`
  - `homework` 的 `insight -1`
  - 作业额外技能奖励再 `-1`
- `4-6`
  - `homework` 的 `insight -2`
  - 作业额外技能奖励归零

#### `work` 透支

影响对象：

- `part_time`

建议档位：

- `0-1`：无惩罚
- `2-3`
  - `coins -2`
  - `fatigue +1`
- `4-6`
  - `coins -4`
  - `fatigue +2`

#### `training` 透支

影响对象：

- `training`

建议档位：

- `0-1`：无惩罚
- `2-3`
  - `aura -1`
  - `mood` 正收益 `-1`
- `4-6`
  - `aura -2`
  - `mood` 正收益归零
  - 额外 `selfControl -1`

### F. 周结算展示改造

当前所有“第一周”文案都需要改为动态周次。

建议：

- 周结算标题：`第 N 周结算`
- 周结算正文：`你以 {rank} 的评定结束了第 {week} 周。当前最强项是 {bestSkillLabel}。`
- 状态栏：`第 N 周结算完成`
- 左侧摘要：显示本周品级、最佳方向、本周主路线、当前长期代价

在最后一周结束时，再增加一层总评：

- 汇总 4 周表现
- 展示路线稳定性
- 展示长期代价演变

### G. 自动实验脚本改造

当前脚本只跑到第一次 `summary`，需要升级为多周版。

固定约束：

- 第 1 周只选一次课
- 后续 3 周沿用同一周课表
- 只更换自由时段填法

建议模板：

- `全课业-4周`
- `全打工-4周`
- `全修炼-4周`
- `均衡混排-4周`
- 可选：`课业+修炼混排-4周`

建议输出结构：

```json
{
  "template": "full-course-4w",
  "weeks": [
    {
      "week": 1,
      "dominantRoute": "study",
      "routeStress": { "study": 0, "work": 0, "training": 0 },
      "rank": "中上品",
      "coins": 14,
      "insight": 18,
      "spirit": 3,
      "aura": 2,
      "fatigue": 6,
      "bestSkill": { "skill": "math", "level": 6 }
    }
  ],
  "final": {}
}
```

每周至少记录：

- `week`
- `dominantRoute`
- `routeStress`
- `rank`
- `coins`
- `insight`
- `spirit`
- `aura`
- `fatigue`
- `bestSkill`
- `buildingStats`

### H. 数值验证目标

期望出现的实验走势：

- `全课业 / 全打工 / 全修炼`
  - 第 1 周具备明显亮点
  - 第 2 周开始出现效率下滑
  - 第 3-4 周继续重复会明显不如均衡路线稳定
- `均衡混排`
  - 第 1 周不一定最强
  - 但第 3-4 周应具备追平或反超能力

建议通过标准：

1. 至少有一条极端路线在第 1 周高于均衡路线。
2. 同一路线连续第 3 周时，周结算品级或关键资源增速明显回落。
3. `均衡混排` 到第 4 周时不应持续落后所有极端路线。

## 实现边界建议

为避免把逻辑缠在一起，建议拆成以下责任块：

- 会话状态扩展：周次、多周历史、长期代价
- 周末收尾逻辑：生成周报、更新 `routeStress`、决定是否进下一周
- 活动结算扩展：读取 `routeStress` 并作用到 `homework / part_time / training`
- 文案与 UI：动态周次、周报文本、最终总评文本
- 自动实验脚本：支持 4 周循环与逐周输出
- 文档同步：把最终规则回写到 [策划案整合.md](/F:/personal/game_t/xianDemo/策划案整合.md)

## 风险与控制

### 风险 1：惩罚来得太早

如果第 1 周后半段就已经感到路线无力，会违背“第一周亮眼”的目标。

控制方式：

- `routeStress` 只在周结算后更新
- 第 1 周内不启用跨周惩罚

### 风险 2：惩罚全部压在周评分，体感不明显

如果只在周结算扣分，玩家会觉得系统暗改结果。

控制方式：

- 惩罚主要作用于下一周的白天活动收益
- 周结算只负责解释，而不是偷偷主导扣分

### 风险 3：均衡路线过于保守导致前期毫无存在感

控制方式：

- 保留极端路线第 1 周的爆发窗口
- 不在第 1 周就给均衡路线过高保底

## 结论

本设计采用“方案 1：周内爆发 + 跨周累积失衡值”：

- 让极端路线在第 1 周保留戏剧性和亮点
- 通过 `routeStress` 让连续 2-4 周重复同一路线产生明显长期代价
- 保留均衡路线的中长期竞争力
- 为后续自动化实验和策划案同步提供统一结构
