# 炼器限时任务小游戏设计

## 目标
在当前多周校园修仙 demo 中新增一套“限时任务小游戏系统”，并以“炼器任务”作为首个落地内容。该系统需要与现有课表、白天结算、多周推进和周总结兼容，同时沉淀为后续符法、道法等小游戏可复用的统一标准。

## 范围
本次设计覆盖以下内容：
- 通用限时任务规则
- 炼器任务的首版玩法规则
- 任务生成、过期、重试、完成奖励的流程接入
- 状态模型、模块拆分、UI 模式与测试策略

本次设计不包含以下内容：
- 多个不同器物配方的完整内容扩展
- 拖拽交互
- 多轮累计锻造
- 二阶段及以上的数值平衡

## 已确认规则
- 采用“限时任务系统”作为统一接法，不把小游戏继续塞进普通 assignment 语义。
- 炼器任务在一周内最后一节 `craft` 课程结算完成后立即解锁。
- 解锁后立即提示，并将“炼器任务”加入自由时段的可排程活动。
- 限时单位按“剩余天数”计算。
- 炼器任务期限固定为 3 天。
- 任务到期未完成则彻底失败并移除。
- 在期限内单次尝试失败后，后续仍可再次排程并重试。
- 每次进入小游戏都从头开始，不保留局内进度。
- 成功奖励采用“克制的数值奖励 + 剧情/周评价标记”。
- 每周最多生成 1 次炼器任务。

## 通用限时任务标准
这套标准将作为后续符法、道法等小游戏的统一模板，具体数值按玩法单独配置。

### 触发规则
- 某类课程在一周内满足任务生成条件后，立即生成对应任务。
- 任务生成时必须触发提示故事或系统提示。
- 任务以独立可排程活动的形式出现，不替换已有课程或普通作业。

### 排程规则
- 玩家必须主动将该任务排入自由时段。
- 任务一旦排入并执行，就会占用当前时段。
- 任务在活动列表中的可见性由任务状态决定；只有 `active` 且未过期的任务才显示。

### 时限规则
- 时限统一使用“剩余天数”表示。
- 每个任务可单独配置期限天数。
- 到期检查在每日结束或次日开始时执行，超过期限即转为失败并移除。

### 尝试规则
- 默认每次进入小游戏都新开一局。
- 单次失败会消耗当前时段，但不直接终止整个任务。
- 只要任务尚未过期，玩家就能再次安排并重试。

### 完成反馈
- 任务完成后同时发放两类收益：
- 轻量数值奖励，例如对应技能、`insight`、`spirit`
- 标记类奖励，例如周评价补充、剧情标志、任务完成记录
- 任务失败后给负面反馈，但不做额外复杂惩罚。

## 炼器任务首版玩法

### 外层任务流程
1. 玩家在一周内正常推进课程流程。
2. 当本周最后一节 `craft` 课程结算完成时，系统生成 `artifact_refining` 任务实例。
3. 系统提示“炼器任务已解锁”，同时将 `炼器任务` 活动加入自由时段候选列表。
4. 玩家在未来 3 天内，可将 `炼器任务` 安排到任意自由时段执行。
5. 进入小游戏后，成功则任务完成并发奖；失败则本时段消耗，但可在期限内重试。
6. 若超过 3 天仍未完成，则任务过期失败并移除。

### 局内规则
- 首版牌阵固定为 3 槽三角阵。
- 一次尝试只结算 1 个三角组合。
- 玩家每次尝试抽取 9 张暗牌。
- 初始可翻开 3 张牌。
- 交互采用“选择卡牌 -> 选择槽位放入”，暂不做拖拽。

### 卡牌类型
- 基础材料：`玄铁`、`灵石`、`木精`
- 功能牌：`观星`、`灵锻`

### 功能牌效果
- `观星`：翻开后立即揭示其上下左右相邻牌。
- `灵锻`：在结算时可视为任意一种基础材料。

### 结算规则
- 三角阵放满 3 张牌后立即结算。
- 分数由“3 张牌组合”决定，而不是单卡固定得分。
- 首版至少要支持显式组合表，例如：
- `玄铁 + 玄铁 + 灵石 = 3 分`
- `玄铁 + 玄铁 + 玄铁 = 1 分`
- 后续组合表继续补充到配置文件，不改变结算结构。

### 成功条件
- 组合分数达到当前器物目标的目标分数。
- 同时满足当前器物目标的最低材料要求。
- 两者同时满足才算任务成功。

### 首版目标建议
- 目标器物固定为 1 个简单目标，例如“聚灵针胚”。
- 目标分数先走低门槛，便于验证流程完整性。
- 最低材料要求首版可设置为“至少包含 1 张玄铁与 1 张灵石”。

## 技术方案

### 数据层
- 新增 `data/tasks.js`
- 负责定义通用任务标准、任务类型配置、炼器目标配置、组合分值表
- 在 `index.html` 中于 `main.js` 之前引入

- 修改 `data/activities.js`
- 新增独立可排程活动 `artifact_refining_task`
- `kind` 使用新类型 `task`
- 该活动默认不常驻显示，由任务系统决定是否可选

- 修改 `data/ui.js`
- 新增任务模式文案、剩余天数、任务状态、按钮文案

- 修改 `data/copy.js`
- 新增解锁、成功、失败、过期、周总结补充文案

### 状态层
在根状态中新增 `tasks`：

```js
tasks: {
  active: [],
  weeklyProgress: {
    craftCompleted: 0,
    craftTotal: 0,
  },
  lastUnlockedTaskStory: null,
}
```

任务实例结构建议如下：

```js
{
  id: "week-1-artifact-refining",
  type: "artifact_refining",
  activityId: "artifact_refining_task",
  status: "active",
  unlockDay: 4,
  expiresOnDay: 7,
  attemptCount: 0,
  successMark: null,
  rewardClaimed: false,
}
```

小游戏运行态建议单独挂在根状态中：

```js
taskRuntime: {
  taskId: null,
  mode: null,
  refining: {
    deck: [],
    slots: [null, null, null],
    revealedCount: 0,
    selectedCardId: null,
    result: null,
  },
}
```

### 领域模块拆分
- 新增 `src/domain/task-system.js`
- 负责通用任务生命周期：
- 记录课程进度
- 判断是否解锁任务
- 控制任务是否可排程
- 判断任务过期
- 处理任务成功/失败结算

- 新增 `src/domain/refining-minigame.js`
- 负责炼器局内玩法：
- 生成牌组与牌面状态
- 翻牌与观星扩展揭示
- 放牌与撤销
- 灵锻替代
- 组合计分与成败判断

- 保持 `src/domain/activity.js` 只处理普通活动结算
- 对 `kind === "task"` 的活动，改为触发任务系统而不是普通 activity 数值结算

### 流程接入点
- 修改 `src/app/day-flow.js`
- 在普通活动结算后，如果活动属于 `craft` 课程，则更新本周 `craft` 进度
- `craftTotal` 可在周开始时根据 `weeklyTimetable` 统计，解锁条件为 `craftCompleted === craftTotal`
- 若该节课结算后首次满足上述条件，则立即解锁炼器任务并推送故事提示
- 若当前时段活动为 `artifact_refining_task`，则切换到 `task` 模式而不是直接完成普通结算

- 修改 `src/app/session.js`
- 初始化 `tasks` 与 `taskRuntime`
- 在确认课表和每次进入新周时重建 `craftTotal`
- 周切换时重置每周进度与当周任务实例

- 修改 `main.js`
- 新增 `task` 模式的渲染分支
- 新增任务界面和炼器界面对应的主面板渲染
- 新增回到日程结算的衔接逻辑

- 修改 `src/app/keyboard-controls.js`
- 为 `task` 模式增加最小键盘控制

### UI 表现
为了贴合当前项目“canvas 做氛围，DOM 做交互”的结构，首版采用以下方式：
- canvas 负责炼器背景和任务氛围
- 主面板 DOM 负责 9 张牌、3 个三角槽位、剩余天数、器物目标、结算按钮
- 不做拖拽，使用点击式放牌

任务可见信息至少包括：
- 任务名
- 剩余天数
- 当前目标器物
- 当前配方需求
- 当前三角阵槽位状态
- 当前尝试结果

## 奖励与失败反馈

### 成功奖励
- `craft` 技能提升
- 少量 `spirit` 或 `insight`
- 周总结中的任务完成标记

### 失败反馈
- 单次失败：只记录一次失败提示，并消耗当前时段
- 到期失败：任务移除，记录一条明确的负面反馈，并在日志或总结中可回看

## 测试策略
至少补以下三组测试：

### 任务系统测试
- 本周最后一节 `craft` 课程后正确解锁炼器任务
- 解锁后活动列表中可见 `炼器任务`
- 3 天内失败后仍可再次排程
- 超过 3 天未完成则任务过期并移除
- 成功后任务不再可排程，且本周不重复生成

### 炼器规则测试
- 9 张牌初始化正确
- 初始翻牌数量正确
- `观星` 揭示上下左右牌
- `灵锻` 可替代缺失材料
- 三角组合按配置表正确计分
- 一次失败后再次进入会重开新局

### 流程接线测试
- 日程执行到任务活动时进入 `task` 模式
- 小游戏结束后能回到日程推进
- 周总结能读取任务完成标记

## 风险与约束
- `main.js` 当前仍承载较多渲染逻辑，任务界面接入时需要控制增量，避免继续堆积无边界 UI 代码。
- `activities` 目前默认以课程/作业/日常活动为主，引入 `task` 类型后需要明确哪些位置允许该类型出现。
- 现有测试覆盖主要围绕周循环和课表，任务系统接入后必须先补规则测试，再写实现。

## 实施建议
先完成通用任务系统和炼器小游戏最小闭环，再扩展更多组合表、更多器物目标和更复杂的交互。首版优先验证“任务生成 -> 排程进入 -> 局内判定 -> 成功/失败 -> 周循环反馈”这一完整链路是否稳定。
