# 玩法实验记录-第三轮

## 本轮目标

验证三路并行改动是否可以同时合入，并复验第二轮后提出的两个方向：

1. `课业 / 打工 / 修炼 / 均衡` 四种路线的周结算差距是否被重新拉开。
2. 自动化实验脚本是否可以稳定重复执行，作为后续数值迭代的基线工具。

## 并行改动冲突检查

本轮实际合入的三组改动如下：

| 代理 | 目标 | 结果 | 写入范围 |
| --- | --- | --- | --- |
| Agent 1 | 活动收益调优 | 已接入 | `data/activities.js` `src/domain/activity.js` |

结论：

- 文件级无直接冲突，三部分写入范围彼此独立。
- 逻辑级存在耦合，但属于预期耦合：
  - 活动收益调优会改变 `coins / insight / aura / skills / fatigue`
  - 周结算公式会重新解释这些数值
  - 自动化脚本负责验证这两个变化叠加后的体感差异
- 当前可以一起保留，不需要拆分回滚。

## 自动化脚本修复

第三个代理产出的脚本最初没有稳定跑完。定位后确认根因是：

- 脚本把静态服务端口写死为 `4173`
- 端口若被旧进程占用，脚本会误连到别的服务
- 那个旧服务退出后，第二个模板开始就会出现 `ERR_CONNECTION_REFUSED`

已修复为：

- 实验脚本改成进程内静态服务
- 端口改为自动分配
- 新增回归测试 `scripts/static-server.test.mjs`

## 结果解读

### 1. 路线差距已经重新出现，但还不够开

- `全打工` 和 `均衡混排` 现在都能稳定到 `中中品`
- `全课业` 与 `全修炼` 落在 `中下品`
- 说明第二轮里“所有路线都挤进高位”的问题已经被压住
- 但 `全打工` 与 `均衡混排` 仍然同档，说明“极端路线”和“均衡路线”的差距还不够明确

### 2. Agent 1 的活动调优起效了

- `课业` 路线的灵感仍然最高，但不再无脑领先到吃掉所有评价维度
- `打工` 路线能拉出极高灵石，但疲劳也显著更高
- `修炼` 路线把优势集中在灵压，没有自动转化成更高周结算

这说明：

- `作业收益递减 + 疲劳惩罚` 已经削弱了课业独大
- 但 `修炼` 路线的周结算映射还偏弱，灵压对周目标的价值感不足

### 3. 记忆拼图 / 建筑玩法本轮还没进入有效验证

自动脚本四条路线的结果里：

- `buildingStats.total` 全部为 `0`
- `bridgeCount` 全部为 `42`

这说明当前自动化脚本虽然能把夜间步骤走完，但还没有拉开“建筑类型”上的差异，因此：

- 本轮结论只适用于日程玩法与周结算玩法
- 还不能用来判断夜间灵块拼接是否影响流派成长

## 合并建议

建议按以下方式合并并继续前进：

1. 保留 Agent 1 的活动调优改动。它已经让课业路线回到可控范围。
2. 保留 Agent 2 的周结算公式。它已经开始奖励多维发展，而不是只吃单轴爆点。
3. 保留并提交 Agent 3 的实验脚本修复。没有稳定脚本，后续数值讨论都会退回手测。

不建议现在回滚任何一组改动，原因是：

- 三组改动一起时，系统终于能稳定跑出“有差异但没完全失控”的结果
- 现在最需要的不是回退，而是继续拉大路线辨识度

## 第四轮复验计划

### A. 单变量目标

只动一个方向：拉开 `全打工` 与 `均衡混排` 的周结算差异。

优先顺序：

1. 先提高“第三维奖励”或均衡奖励权重
2. 若仍不足，再略压 `coins` 在周结算中的权重
3. 暂不继续动 `homework`，避免把上一轮刚稳定下来的课业节奏再次打乱

### B. 验证样本

继续保留四条固定模板：

- 全课业
- 全打工
- 全修炼
- 均衡混排

新增一个补充模板：

- `修炼+课业混排`，用于检验“学业辅助修炼”是否能形成比纯修炼更合理的中间路线

### C. 通过标准

满足以下三条即可认为第四轮有效：

1. `均衡混排` 稳定高于 `全打工` 一个档位，或至少在同档下拥有明显更强的事件/技能/建筑收益。
2. `全修炼` 不再稳定垫底，能通过灵压或夜间建筑体系获得明确补偿。
3. 夜间建筑统计不再全部相同，至少能看出不同路线带来的灵块类型差异。

### D. 需要补的工具能力

在第四轮前，建议优先增强自动实验脚本：

1. 记录每天拿到的灵块类型分布。
2. 记录夜间放置后形成的建筑类型数量。
3. 记录白天随机事件触发次数与来源。
4. 允许模板配置“优先放哪种建筑”，否则夜间玩法始终只能做通关型验证，不能做流派型验证。
